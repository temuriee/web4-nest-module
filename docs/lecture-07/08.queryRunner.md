## 🔹 1. რა არის `QueryRunner`?

👉 **`QueryRunner` არის TypeORM-ის ყველაზე დაბალი დონის ობიექტი, რომელიც გაძლევს პირდაპირ კონტროლს მონაცემთა ბაზასთან.**

თუ ჩვეულებრივ იყენებ:

```ts
this.userRepository.save(user);
```

ეს მიდის TypeORM-ის _EntityManager_-ზე, რომელიც თავის მხრივ _DataSource_-ის საშუალებით ქმნის _QueryRunner_-ს "შიგნით" — შენ ამას არ ხედავ.

მაგრამ როცა **გჭირდება სრული კონტროლი** (მაგალითად, ტრანზაქციები, custom query-ები ან მრავალი ოპერაციის ერთიანად შესრულება),
შენ შეგიძლია შექმნა შენი საკუთარი `QueryRunner` ხელით.

---

## 🔹 2. რატომ ვქმნით ხელით?

როდესაც შენ აკეთებ:

```ts
const queryRunner = this.dataSource.createQueryRunner();
```

ეს ნიშნავს:

> “მომეცი ახალი, ცალკე **კავშირი (connection)** მონაცემთა ბაზასთან, რომ მასზე მე თვითონ ვმართო რა ხდება”.

- TypeORM ჩვეულებრივ იყენებს **კავშირის პულს (connection pool)** — ანუ რამდენიმე მზად კავშირს DB-სთან.
- `QueryRunner` ერთ-ერთ ასეთ კავშირს იღებს დროებით და შენ გაძლევს პირდაპირ წვდომას მასზე.
- ამით შეგიძლია **გააკონტროლო transaction-ი**, **გააკეთო query-ები**, **შეასრულო მრავალი ნაბიჯი ერთდროულად**, და ბოლოს **თავისუფლად გაუშვა ეს კავშირი**.

---

## 🔹 3. მისი სიცოცხლის ციკლი (Lifecycle)

მოდით ვნახოთ, რა ხდება ეტაპობრივად:

```ts
const queryRunner = this.dataSource.createQueryRunner(); // ①
await queryRunner.connect(); // ②
await queryRunner.startTransaction(); // ③
try {
  // do operations
  await queryRunner.commitTransaction(); // ④
} catch (err) {
  await queryRunner.rollbackTransaction(); // ⑤
} finally {
  await queryRunner.release(); // ⑥
}
```

🧩 ეტაპები:

| ნაბიჯი                      | აღწერა                                                                                                                  |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **① createQueryRunner()**   | ქმნის QueryRunner ობიექტს — ჯერ უბრალოდ ცარიელია, კავშირი არ აქვს                                                       |
| **② connect()**             | ქმნის რეალურ ფიზიკურ კავშირს DB-სთან (1 კონექშენი pool-იდან)                                                            |
| **③ startTransaction()**    | იწყებს ახალ ტრანზაქციას — ყველაფერი შემდეგი `INSERT`, `UPDATE`, `DELETE` მხოლოდ მაშინ დაფიქსირდება, როცა commit მოხდება |
| **④ commitTransaction()**   | ყველა ოპერაცია დამტკიცებულია DB-ში — ახლა ისინი რეალურად შენახულია                                                      |
| **⑤ rollbackTransaction()** | თუ შეცდომა მოხდა, ყველა ცვლილი ოპერაცია გაუქმდება, როგორც “არაფერიც არ მომხდარა”                                        |
| **⑥ release()**             | აბრუნებს გამოყენებულ კავშირს pool-ში (მნიშვნელოვანია, ყოველთვის გამოიძახო!)                                             |

## 🔹 4. QueryRunner-ს აქვს საკუთარი Manager (EntityManager)

შენ იყენებ:

```ts
queryRunner.manager.create(User, user);
queryRunner.manager.save(newUser);
```

ეს `manager` არის **EntityManager**, რომელიც **ამ QueryRunner-ზეა მიბმული**.

> განსხვავებით ჩვეულებრივი `this.userRepository`-სგან, რომელიც იყენებს default EntityManager-ს (shared connection),
> `queryRunner.manager` იყენებს _შენ მიერ შექმნილ transaction-ს_.

ანუ ყველაფერი, რაც აქ გააკეთე — ეკუთვნის იმ კონკრეტულ transaction-ს.

---

## 🔹 5. QueryRunner-ის შესაძლებლობები

### ✅ ა) transaction კონტროლი

შეგიძლია მართო მრავალი ოპერაცია ისე, რომ ან ყველაფერი ჩაიწეროს ან არაფერი:

```ts
await queryRunner.startTransaction();
try {
  await queryRunner.manager.save(user);
  await queryRunner.manager.save(profile);
  await queryRunner.commitTransaction();
} catch (e) {
  await queryRunner.rollbackTransaction();
}
```

---

### ✅ ბ) raw SQL query-ები

შეგიძლია პირდაპირ SQL გაუშვა:

```ts
const users = await queryRunner.query(`SELECT * FROM users WHERE age > $1`, [
  18,
]);
```

---

### ✅ გ) schema-level ოპერაციები

შეგიძლია ცვალო ცხრილები, სვეტები და ა.შ.:

```ts
await queryRunner.createTable(...);
await queryRunner.addColumn(...);
```

(მხოლოდ მაშინ, როცა migrations-ს ან schema manipulation-ს აკეთებ)

---

### ✅ დ) nested transactions (savepoints)

შეგიძლია პატარა შიდა ტრანზაქციები შექმნა ერთი დიდი ტრანზაქციის შიგნით:

```ts
await queryRunner.startTransaction();
try {
  // ოპერაცია 1
  await queryRunner.commitTransaction();
} catch {
  await queryRunner.rollbackTransaction();
}
```

ან savepoint-ებით იმუშაო:

```ts
await queryRunner.createSavepoint('step1');
// ...
await queryRunner.rollbackTransactionToSavepoint('step1');
```

## 🔹 6. რატომ არ ვიყენებთ QueryRunner ყველა შემთხვევაში?

მიუხედავად იმისა, რომ QueryRunner ძლიერია, მას აქვს ხარჯები:

- ის იღებს **ერთ სრულ კონექშენს** pool-იდან → თუ არ გამოუშვებ (`release()`), შეიძლება სხვა კოდმა ვერ მიიღოს კავშირი.
- ტრანზაქციები უფრო ნელა მუშაობენ, ვიდრე უბრალო ოპერაციები.
- მის გარეშე CRUD გაცილებით მარტივი და automanaged-ია.

ამიტომ QueryRunner ვიყენებთ მხოლოდ მაშინ, როცა გვჭირდება:

- ბევრი ოპერაციის ერთდროულად შესრულება, რომლებიც ან ყველა წარმატებით ჩაიწეროს ან არცერთი;
- სპეციფიკური კონტროლი DB ოპერაციებზე;
- custom raw SQL ან schema manipulation.

```

│             DataSource               │
│ ─────────────────────────────────── │

```

## 🔹 8. პატარა რეალური მაგალითი

```ts
@Injectable()
export class OrdersService {
  constructor(private readonly dataSource: DataSource) {}

  async createOrder(orderDto: CreateOrderDto) {
    const queryRunner = this.dataSource.createQueryRunner();
    await queryRunner.connect();
    await queryRunner.startTransaction();

    try {
      const order = queryRunner.manager.create(Order, orderDto);
      await queryRunner.manager.save(order);

      const payment = queryRunner.manager.create(Payment, {
        orderId: order.id,
        amount: orderDto.amount,
      });
      await queryRunner.manager.save(payment);

      await queryRunner.commitTransaction();
      return order;
    } catch (error) {
      await queryRunner.rollbackTransaction();
      throw new InternalServerErrorException('Could not create order');
    } finally {
      await queryRunner.release();
    }
  }
}
```

**თუ `payment` ვერ ჩაიწერება, order-იც გაუქმდება.**
ანუ შენს ბაზაში არასდროს დარჩება “გადაუხდელი შეკვეთა” შემთხვევით.

## 🔹 1. `QueryRunner` რეალურად ყოველთვის მუშაობს “ფარდის მიღმა” 🤫

როდესაც შენ ჩვეულებრივ წერ რაღაც ასეთს:

```ts
await this.userRepository.save(user);
```

👉 TypeORM შიგნით მაინც იყენებს `QueryRunner`-ს —
უბრალოდ **შენ არ იძახებ მას ხელით**, TypeORM კი **ავტომატურად ქმნის, აერთებს და ათავებს** მის გამოყენებას.

ანუ ყოველ ჯერზე, როცა შენ აკეთებ:

- `.save()`
- `.find()`
- `.delete()`
- `.update()`
- `.query()`

TypeORM შიგნით:

1. იღებს **ერთ connection-ს pool-იდან**,
2. ქმნის შიდა `QueryRunner` ობიექტს,
3. უშვებს SQL-ს,
4. და ბოლოს `release()`-ს აკეთებს ავტომატურად.

შენ ამას ვერ ხედავ, მაგრამ ეს ხდება ყოველთვის — ამიტომ “რეალურად queryRunner ყოველთვის მუშაობს”, როგორც შენ ზუსტად თქვი ✅

---

## 🔹 2. როდის **არ უნდა** შექმნა შენით

როდესაც შენ უბრალოდ აკეთებ ერთ ან ორ დამოუკიდებელ ოპერაციას, მაგალითად:

```ts
await this.postsRepository.save(post);
await this.tagsRepository.save(tag);
```

TypeORM თვითონ იფიქრებს:

> “არაა საჭირო ცალკე ტრანზაქცია — ყოველი ეს ქმედება ცალ-ცალკე queryRunner-ით დაასრულებ”.

ამიტომ ხელით `QueryRunner` შექმნა ზედმეტია — overhead-ს აჩენს:

- თითოეულ QueryRunner-ს თავისი connection უნდა,
- release თუ დაგავიწყდა → pool გაიჭედება,
- კოდი ხდება უფრო გრძელი და რთულად წასაკითხი.

---

## 🔹 3. როდის გჭირდება **ხელით შექმნა**

მხოლოდ მაშინ, როცა გინდა, რომ რამდენიმე ოპერაცია:

- **ერთიანად** შესრულდეს, ან
- **ერთიანად გაუქმდეს** თუ შეცდომა მოხდება,
  ან როცა გჭირდება **custom query-ების** შესრულება იმავე transaction-ში.

მაგალითად:

```ts
// თუ გინდა user და post ერთდროულად ჩაიწეროს
// და თუ რომელიმე ჩავარდა, ორივე rollback-დეს:
const queryRunner = this.dataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();

try {
  const user = await queryRunner.manager.save(User, userData);
  const post = await queryRunner.manager.save(Post, { ...postData, user });
  await queryRunner.commitTransaction();
} catch (e) {
  await queryRunner.rollbackTransaction();
} finally {
  await queryRunner.release();
}
```

თუ შეცდომა მოხდა `post`-ზე — **user-იც აღარ ჩაიწერება**.
ეს შეგიძლია მხოლოდ მაშინ, როცა შენ თვითონ მართავ `QueryRunner`-ს.

## 🔹 4. კიდევ ერთი შემთხვევა — როცა გჭირდება უფრო დაბალი დონე

ზოგჯერ გინდა:

- გამოიყენო **raw SQL query** (არა repository API),
- მართო **schema ცვლილებები** runtime-ზე,
- ან იმუშაო **migration-ის** დონეზე (მაგალითად, ცხრილების დამატება, წაშლა),
- ან გააკეთო **savepoint-ები** (nested transactions).

მაგალითად:

```ts
const queryRunner = this.dataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();

try {
  await queryRunner.query(
    'UPDATE accounts SET balance = balance - 100 WHERE id = $1',
    [1],
  );
  await queryRunner.query(
    'UPDATE accounts SET balance = balance + 100 WHERE id = $2',
    [2],
  );
  await queryRunner.commitTransaction();
} catch (e) {
  await queryRunner.rollbackTransaction();
} finally {
  await queryRunner.release();
}
```

ეს უკვე ისეთი custom მოქმედებაა, რასაც Repository ვერ გააკეთებს ისე ზუსტად.

| ერთი `.save()` ან `.find()`        | ❌ არა                    | TypeORM ქმნის და ხურავს ავტომატურად  |

| Transaction (ყველაფერი ან არაფერი) | ✅ დიახ                   | შენ ქმნი და მართავ queryRunner-ს     |
| Raw SQL query იგივე transaction-ში | ✅ დიახ                   | QueryRunner გაძლევს პირდაპირ წვდომას |
| Schema ან migration manipulation   | ✅ დიახ                   | მხოლოდ QueryRunner-ს შეუძლია         |

## 🔹 6. როგორ ფიქრობდეს ამაზე მარტივად 💡

შეგიძლია ასე დაიმახსოვრო:

> TypeORM ყოველ ჯერზე “ხელს იყენებს” queryRunner-ს,
> მაგრამ შენ “მთელ სხეულს” (control-ს) იღებ მხოლოდ მაშინ, როცა თვითონ გჭირდება ყველაფერი მართო.

ანუ ჩვეულებრივ შენს ადგილას TypeORM აკეთებს:

```
create runner → run query → release runner
```

ხოლო როცა transaction გჭირდება, შენ აკეთებ ზუსტად იმავე რამეს, უბრალოდ **გააგრძელებ runner-ს ცოცხლად**, სანამ ყველაფერი არ შესრულდება.

## 🔹 7. კარგი პრაქტიკა (შენთვის როგორც senior დონის ჩვევები)

1. **ყოველთვის გამოუშვი (`release()`) runner**, თუნდაც შეცდომა მოხდეს (`finally` ბლოკში).
2. არასდროს შექმნა **ბევრი QueryRunner ერთდროულად** — თითო runner იღებს ერთ კავშირს pool-იდან.
3. გამოიყენე **`this.dataSource.transaction()`** როცა შეგიძლია —
   ეს wrapper-ია, რომელიც ავტომატურად ქმნის, მართავს და ხურავს runner-ს შენ მაგივრად.

მაგალითად ასე:

```ts
await this.dataSource.transaction(async (manager) => {
  const user = await manager.save(User, userData);
  const post = await manager.save(Post, { ...postData, user });
});
```

ეს იგივეა რაც ხელით runner-ის მართვა, უბრალოდ ბევრად სუფთად.
