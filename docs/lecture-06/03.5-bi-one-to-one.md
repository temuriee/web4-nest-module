## 🎯 Bi-directional One-to-One მაგალითი

ვიყენებთ ისევ იმავე ისტორიას —
🧍‍♂️ `User` (მოსწავლე) და 🪪 `Profile` (პროფილი)

მაგრამ ამჯერად **ორივე მხარე** “იცნობს” ერთმანეთს.

---

### 🧩 კოდი:

```ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  OneToOne,
  JoinColumn,
} from 'typeorm';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  // 🧠 User-ს ჰყავს ერთი პროფილი
  @OneToOne(() => Profile, (profile) => profile.user)
  @JoinColumn() // აქ ინახება foreign key
  profile: Profile;
}

@Entity()
export class Profile {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  age: number;

  // 🧠 პროფილიც იცის, რომ ეკუთვნის იუზერს
  @OneToOne(() => User, (user) => user.profile)
  user: User;
}
```

---

### 📘 ახსნა ნაბიჯ-ნაბიჯ

1. `User`-ში გვაქვს:

   ```ts
   @OneToOne(() => Profile, (profile) => profile.user)
   @JoinColumn()
   profile: Profile;
   ```

   👉 ეს ნიშნავს: “მე (User) მაქვს ერთი Profile, და მე ვარ ის მხარე, სადაც foreign key შეინახება.”

2. `Profile`-ში გვაქვს:

   ```ts
   @OneToOne(() => User, (user) => user.profile)
   user: User;
   ```

   👉 ეს ნიშნავს: “მე ვარ Profile და ვიცი, რომ ვეკუთვნი ამ კონკრეტულ User-ს.”

3. ⚙️ შედეგი მონაცემთა ბაზაში:
   - `user` ცხრილში იქნება foreign key `profileId` (ან როგორც დაარქმევ),
   - და ორივე entity-ს შეუძლია ერთმანეთზე წვდომა.

---

### 🧠 როგორ მუშაობს პრაქტიკაში

```ts
// შექმნა
const profile = new Profile();
profile.age = 16;

const user = new User();
user.name = 'ლევანი';
user.profile = profile;

await dataSource.manager.save(profile);
await dataSource.manager.save(user);

// წვდომა
const foundUser = await userRepository.findOne({
  where: { id: 1 },
  relations: ['profile'], // profile-საც წამოიღებს
});

console.log(foundUser.profile.age); // → 16

const foundProfile = await profileRepository.findOne({
  where: { id: 1 },
  relations: ['user'],
});

console.log(foundProfile.user.name); // → ლევანი
```

### 🔄 შეჯამება:

| ტიპი                | აღწერა                       | ვის აქვს ინფორმაცია |
| ------------------- | ---------------------------- | ------------------- |
| **Uni-directional** | ერთი მხარე იცის მეორე        | მხოლოდ `User`       |
| **Bi-directional**  | ორივე მხარე იცნობს ერთმანეთს | `User` ↔ `Profile` |

## 🧩 ნაწილი 1:

👉 **`@OneToOne(() => Profile)`** — რას აკეთებს `Profile` აქ?

ეს არ აბრუნებს მთელ ცხრილს (table-ს),
მაგრამ **TypeORM-ს ეუბნება**, რომ “ეს ურთიერთობა ეხება `Profile` entity-ს”.

ანუ სინტაქსურად:

```ts
@OneToOne(() => Profile)
```

ეს არის ფუნქცია, რომელიც **უბრალოდ აბრუნებს class-ს**, ანუ `Profile`-ს.
ეს TypeORM-ს ეხმარება ისწავლოს —
**რომელი entity-სთან გვაქვს კავშირი**, _მაგრამ არ ქმნის მონაცემებს ან ცხრილს პირდაპირ._

---

### მარტივად რომ ვთქვათ:

- ❌ არ აბრუნებს მონაცემებს
- ❌ არ აკეთებს query-ს
- ✅ უბრალოდ ამბობს: “მეორე მხარე არის `Profile` entity”

ეს იმისთვისაა, რომ TypeORM-მა იცოდეს როგორ ააგოს foreign key და როგორ “დააკავშიროს” ORM-შიგნით მოდელები ერთმანეთთან.

---

## 🧩 ნაწილი 2:

👉 ახლა ვნახოთ ეს ვარიანტი:

```ts
@OneToOne(() => Profile, (profile) => profile.user)
```

აქ მეორე არგუმენტი `(profile) => profile.user`
უკვე დაკავშირებულია **bi-directional ურთიერთობასთან**.

### ანუ რას ნიშნავს ეს?

- პირველი არგუმენტი `() => Profile` → ამბობს **რომელ entity-ზეა კავშირი**
- მეორე არგუმენტი `(profile) => profile.user` → ამბობს **მეორე მხარეს რომელი ველი უკავშირდება ამ კავშირს**

ეს მეორე არგუმენტი _აუცილებელია მხოლოდ მაშინ_, როცა გინდა რომ კავშირი **ორი გზით იმუშაოს**
(ანუ ორივე entity-მ იცოდეს ერთმანეთზე).

---

### 📘 შედარება Uni vs Bi

#### Uni-directional:

```ts
@OneToOne(() => Profile)
@JoinColumn()
profile: Profile;
```

🧠 მხოლოდ `User`-ს აქვს ინფორმაცია `Profile`-ზე.
TypeORM-ს არ სჭირდება იცოდეს მეორე მხარეს რა ჰქვია შესაბამის ველს.

#### Bi-directional:

```ts
@OneToOne(() => Profile, (profile) => profile.user)
@JoinColumn()
profile: Profile;
```

🧠 ახლა ორივე მხარე იცნობს ერთმანეთს.
ამიტომ TypeORM უნდა იცოდეს:

> "Profile-ის შიგნით რომელი ველია, რომელიც `User`-ს უკავშირდება?"

და ამიტომაც ვწერთ `(profile) => profile.user`.

---

## 💡 თუ მეორე მხარეს (ანუ `Profile`-ში) არ გაქვს `user` ველი,

მაშინ `(profile) => profile.user`–ის ჩაწერა **შეუძლებელია და არც საჭიროა**.
ეს გამოიყენება მხოლოდ მაშინ, როცა ორივე მხარეს გაქვს ურთიერთობა აღწერილი.

---

## 🧠 მოკლედ შეჯამება:

| სინტაქსი                                              | რისთვისაა       | საჭიროა თუ არა                                   |
| ----------------------------------------------------- | --------------- | ------------------------------------------------ |
| `@OneToOne(() => Profile)`                            | Uni-directional | ✅ დიახ                                          |
| `@OneToOne(() => Profile, (profile) => profile.user)` | Bi-directional  | ✅ საჭიროა მეორე მხარის ველი არსებობდეს (`user`) |

---

მოკლედ:

- პირველი არგუმენტი — ეუბნება TypeORM-ს _რომელი entity-ა მეორე მხარე_
- მეორე არგუმენტი — ეუბნება _მეორე მხარეს რომელი ველი უკავშირდება ამ კავშირს_

> ✅ **კი**, მეორე არგუმენტში არსებული ცვლადი (`profile`) წარმოადგენს **იმ Entity-ს**, რომელსაც პირველ არგუმენტში გადასცემ `@OneToOne(() => Profile)`.

---

მოდი ავხსნათ ნაბიჯ-ნაბიჯ 👇

---

## 🧩 არგუმენტების ანატომია

ანოტაცია:

```ts
@OneToOne(() => Profile, (profile) => profile.user)
```

შეიცავს **ორგანოებს** 😄
1️⃣ `() => Profile` → ეს ფუნქცია აბრუნებს **Entity class-ს**, რომლისკენაც გვაქვს ბმა.
2️⃣ `(profile) => profile.user` → ეს არის **callback**, სადაც TypeORM ამ `Profile` entity-ს მოიაზრებს როგორც ცვლადს —
და ჩვენ ვეუბნებით: “აი, ამ entity-ში რომელი ველია უკავშირდება ამ კავშირს?”

---

### 👉 ანუ ამ მაგალითში:

```ts
@OneToOne(() => Profile, (profile) => profile.user)
```

TypeORM კითხულობს ასე:

> “ეს ურთიერთობა ეხება `Profile` entity-ს,
> და მეორე მხარეს შესაბამისი ველია `user`.”

აქ:

- `profile` უბრალოდ **ლოკალური ცვლადია**,
- მაგრამ მისი ტიპი — `Profile`,
- ამიტომ TypeORM-ს შეუძლია “profile.user” სტრუქტურის გააზრება.

---

## 🧠 მარტივი ვიზუალიზაცია

```ts
@Entity()
export class User {
  @OneToOne(() => Profile, (profile) => profile.user)
  @JoinColumn()
  profile: Profile;
}

@Entity()
export class Profile {
  @OneToOne(() => User, (user) => user.profile)
  user: User;
}
```

აქ `profile` მეორე არგუმენტში → ნიშნავს **Profile entity-ს ინსტანცას**,
და შენ ეუბნები TypeORM-ს: “Profile-ს აქვს ველი user”.

---

## ⚙️ სხვა მაგალითი შენი კოდის ფორმატით:

```ts
@OneToOne(() => Post, (profile) => profile.user)
```

🧠 აქ კი:

- პირველი არგუმენტი — `() => Post` — ნიშნავს: "მეორე მხარე არის Post entity".
- მეორე არგუმენტი — `(profile) => profile.user` — ახლა `profile` უბრალოდ **სახელია**, შეგიძლია დაარქვა სხვანაირადაც:

```ts
@OneToOne(() => Post, (post) => post.user)
```

ზუსტად იგივეა!
TypeORM გაიგებს, რომ:

> “ეს ურთიერთობა ეხება `Post` entity-ს,
> და მეორე მხარეს შესაბამისი ველია `user`.”

---

### ✅ ანუ, დიახ:

- თუ პირველ არგუმენტში გადასცემ `Profile` → მეორე callback-ის ცვლადი წარმოადგენს `Profile`-ს.
- თუ გადასცემ `Post` → მეორე callback-ის ცვლადი წარმოადგენს `Post`-ს.
- ცვლადის სახელი (`profile`, `post`, `anything`) მნიშვნელობა **არ აქვს** —
  მთავარია, რომ მის სტრუქტურაში სწორი ველი მიუთითო (`.user`).
