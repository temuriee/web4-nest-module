## 🎯 იდეა მარტივად:

### 🧱 Cascade Delete

> როცა ერთი ჩანაწერის წაშლისას, **ავტომატურად იშლება მასზე დამოკიდებული ჩანაწერებიც.**

### 🌙 Soft Delete

> როცა ჩანაწერი თითქოს იშლება, მაგრამ სინამდვილეში **ბაზაში რჩება** — უბრალოდ აღინიშნება როგორც “წაშლილი”.

---

## 💡 მაგალითი ისტორიით:

ვიღოთ:

- 👩‍🏫 **Teacher**
- 👦 **Student**

ერთ მასწავლებელს ჰყავს ბევრი მოსწავლე (One-to-Many).

---

### 🧱 **Cascade Delete**

თუ მასწავლებელს წაშლი → მასზე მიბმული ყველა მოსწავლე **ბაზიდან წაიშლება სრულად.**

```ts
@OneToMany(() => Student, (student) => student.teacher, {
  cascade: true,      // ავტომატური ცვლილებები
  onDelete: "CASCADE" // წაშლისას cascade delete
})
students: Student[];
```

🧩 შედეგი:

- წაშლი `Teacher (id:1)`
- TypeORM ავტომატურად წაშლის **ყველა** `Student`, სადაც `teacher_id = 1`

💥 მონაცემები სრულად ქრება (აღარ არსებობს ბაზაში).

---

### 🌙 **Soft Delete**

Soft delete სხვა იდეაა —
ის **არ შლის** რეალურად ჩანაწერს.
უბრალოდ აყენებს დროშას: `"deletedAt"` ან `"isDeleted" = true"`.

```ts
@Entity()
export class Student {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  @DeleteDateColumn() // 👈 TypeORM სპეციალური სვეტი
  deletedAt: Date; // როცა არ არის null → ნიშნავს წაშლილია
}
```

📘 ახლა, როცა წაშლი:

```ts
await studentRepository.softRemove(student);
```

👉 TypeORM უბრალოდ ჩაწერს:

| id  | name | deletedAt  |
| --- | ---- | ---------- |
| 1   | Lika | 2025-10-28 |

🧠 მაგრამ ჩანაწერი **ჯერ კიდევ არსებობს**, უბრალოდ “დამალულია”.

---

## ⚖️ შედარების ცხრილი

| ფუნქცია               | **Cascade Delete**                   | **Soft Delete**                                  |
| --------------------- | ------------------------------------ | ------------------------------------------------ |
| რა ხდება წაშლისას     | შლიან რეალურად                       | არ შლიან, მხოლოდ ნიშნავენ                        |
| მონაცემი ბაზაში       | ქრება                                | რჩება                                            |
| აღდგენა შესაძლებელია? | 🚫 არა                               | ✅ კი (restore)                                  |
| მიზანი                | დამოკიდებული ჩანაწერების გასუფთავება | უსაფრთხო არქივირება                              |
| სად გამოიყენება       | მაგალითად: User → Posts              | მაგალითად: Orders, Users (როცა ისტორია გჭირდება) |

---

## 🎯 ტიპური გამოყენება

- 🧱 **Cascade Delete:**
  როცა გინდა, რომ შვილობილი მონაცემები **მთავართან ერთად გაქრეს**
  (მაგ. წაშლი მომხმარებელს და მისი ყველა კომენტარი ქრება)

- 🌙 **Soft Delete:**
  როცა გინდა “წაშლა” მაგრამ რეალურად შეინარჩუნო
  (მაგ. მომხმარებელი შლის თავის პროფილს, მაგრამ ადმინისტრატორს უნდა ჰქონდეს ჩანაწერი)

---

## ⚙️ TypeORM-ში განსხვავებული ბრძანებები:

| მოქმედება     | ბრძანება                        | აღწერა                     |
| ------------- | ------------------------------- | -------------------------- |
| რეალური წაშლა | `repository.remove(entity)`     | რეალურად შლის              |
| რბილი წაშლა   | `repository.softRemove(entity)` | ნიშნავს როგორც წაშლილს     |
| აღდგენა       | `repository.restore(id)`        | უკან აბრუნებს Soft-წაშლილს |

---

## 🎨 ანალოგია ცხოვრებიდან:

- **Cascade Delete** — როგორც როცა მშობელი LEGO-ს სახლი დაანგრიე და ყველა პატარა ნაწილიც დაიშალა 🧱💥
- **Soft Delete** — როცა LEGO-ს სახლი კარადაში შენახე და უბრალოდ აღარ ჩანს, მაგრამ შეგიძლია ისევ ააწყო 🌙✨
