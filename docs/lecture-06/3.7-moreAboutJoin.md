## 🧩 ჯერ ერთი ტერმინი დავამყაროთ

**Owning table (ანუ "მფლობელი მხარე")** —
ეს არის ის entity, **რომელიც შეიცავს foreign key-ს**
და ამით “ფლობს” ურთიერთობას.

ანუ:

```ts
@OneToOne(() => Profile)
@JoinColumn()  // 👉 აქ foreign key ინახება
profile: Profile;
```

ამ შემთხვევაში `User` არის **owning side**, რადგან foreign key `profileId` ინახება `users` ცხრილში.

---

## 🎯 ახლა მთავარი კითხვა:

> რა უპირატესობა აქვს Owning Table-ს?

აი ძირითადი პუნქტები მარტივი ენით 👇

---

### 1️⃣ ის "კონტროლს" ფლობს ურთიერთობაზე

Owning table არის **ერთი ერთადერთი მხარე**,
რომელმაც შეიძლება პირდაპირ შეცვალოს, დაამატოს ან წაშალოს ურთიერთობა.

🔹 მაგ.:
როცა შენ `user.profile = profile` დაწერ და `save(user)` გააკეთებ,
TypeORM სწორედ `User`-ში ჩაანაცვლებს `profileId`–ს —
**არ წავა Profile-ის ცხრილში ცვლილებების შესატანად**.

თუ კი `Profile` არ არის owning side,
მისი `.user = user` შეცვლა **არ იმუშავებს**,
სანამ ცვლილება owning მხარეზე არ გაკეთდება.

🧠 ანუ:

> მხოლოდ owning მხარეს შეუძლია გადაწყვიტოს, ვინ ვის ეკუთვნის.

---

### 2️⃣ მხოლოდ Owning table ქმნის foreign key-ს ბაზაში

შენ რომ არ მიუთითო `@JoinColumn()` მეორე მხარეს,
TypeORM ზუსტად “ფლობს” კავშირს იმ მხარე, სადაც FK შეინახა.

🧱 ანუ:

- მხოლოდ მას აქვს ფიზიკური foreign key column
- ამიტომ მასზეა დამოკიდებული referential integrity (ბმების სისწორე)

---

### 3️⃣ უფრო მარტივი query–ებია და მენეჯმენტი

რადგან foreign key მის ცხრილშია,
შენ შეგიძლია მარტივად გააკეთო:

```ts
await userRepository.find({
  relations: ['profile'],
});
```

ან პირდაპირ filter–ი foreign key–ზე:

```ts
await userRepository.find({
  where: { profile: { id: 3 } },
});
```

თუ კი Profile იქნებოდა owning side,
მაშინ უნდა წასულიყავი მეორე entity-ში და იქიდან მოგემართა.

---

### 4️⃣ კავშირი უფრო "ლოგიკურად" იკითხება

ჩვეულებრივ One-to-One სიტუაციაში,
foreign key ტარდება იმ entity-ში,
**ვისი ქონებაც უფრო ბუნებრივია**.

მაგალითად:

| სცენარი              | ვისთან ჯობია FK      | რატომ                        |
| -------------------- | -------------------- | ---------------------------- |
| `User` ↔ `Profile`  | `User` (owning side) | პროფილი მიეკუთვნება იუზერს   |
| `Car` ↔ `Engine`    | `Car`                | ძრავი ეკუთვნის მანქანას      |
| `Employee` ↔ `Desk` | `Desk`               | მაგიდა ეკუთვნის თანამშრომელს |

🧠 ანუ FK “მიდის” იმ მხარეს, ვინც რეალურად _ფლობს_ მეორე ობიექტს.

---

### 5️⃣ აუცილებელია კავშირების სინქრონიზაციისას

თუ ორმხრივი (bi-directional) ურთიერთობა გაქვს,
TypeORM-მა უნდა იცოდეს **რომელი მხარეა owning**,
რათა ბაზაში ერთი Foreign Key არ “გაორმაგდეს”.

`@JoinColumn()` სწორედ ამიტომ იწერება მხოლოდ ერთ მხარეს —
ეს მიანიშნებს TypeORM-ს:

> "აი ეს მხარეა ძირითადი. მასში შეინახე foreign key."

---

## 📘 შეჯამება

| მახასიათებელი                       | Owning Table (FK აქვს) | Non-owning Table                 |
| ----------------------------------- | ---------------------- | -------------------------------- |
| ინახავს foreign key-ს               | ✅ კი                  | ❌ არა                           |
| მართავს ურთიერთობას                 | ✅ კი                  | ❌ ვერ ცვლის პირდაპირ            |
| ქმნის FK ბაზაში                     | ✅ კი                  | ❌ არა                           |
| შეიძლება გამოიყენოს `@JoinColumn()` | ✅ კი                  | ❌ არა                           |
| მუშაობს `save()`-ზე პირდაპირ        | ✅ კი                  | ❌ არა (საჭიროა owner-ის შეცვლა) |

---

## 💬 დასამახსოვრებელი ფრაზა

> “Owning side არის როგორც ურთიერთობის ‘ბოსი’ —
> მას აქვს გასაღები (foreign key),
> და მხოლოდ ის წყვეტს, ვის ეკუთვნის მეორე მხარე.”
