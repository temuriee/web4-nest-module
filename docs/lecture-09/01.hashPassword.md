### 🧱 რა ხდება აქ საერთოდ?

შენ გაქვს ორი ფაილი:

1. **hashing.provider.ts** → ეს არის _აბსტრაქტული კლასი_ (ანუ მხოლოდ "გეგმა")
2. **bcrypt.provider.ts** → ეს არის _ნამდვილი შესრულება_ (აქจริงად ხდება hash / compare)

### 💭 წარმოიდგინე ასეთი მაგალითი:

შენი სკოლა ამბობს:

> ვინც იქნება მასწავლებელი, აუცილებლად უნდა ქონდეს ორი ფუნქცია:
>
> - ასწავლოს (`teach()`)
> - შეაფასოს (`grade()`)

ეს არის **წესი**, ანუ გეგმა, მაგრამ არა ნამდვილი ადამიანი.

ახლა მოდის ორი მასწავლებელი:

- ნინო მასწავლებელი → ასწავლის მუსიკას
- გიორგი მასწავლებელი → ასწავლის მათემატიკას

ორივე ემორჩილება იმ ერთსა და იმავე წესს (გეგმას), მაგრამ თავისი გზით ასრულებს.

Nest-ში ეგეთი რაღაც ხდება 👇

### 🧩 Hashing Provider გეგმაა

```ts
export abstract class HashingProvider {
  abstract hash(value: string): Promise<string>;
  abstract compare(value: string, hashedValue: string): Promise<boolean>;
}
```

ეს არ აკეთებს არაფერს — უბრალოდ ამბობს:

> ვინც HashingProvider იქნება, ვალდებულია ჰქონდეს `hash` და `compare` მეთოდები.

---

### 🔐 Bcrypt Provider არის ნამდვილი შემსრულებელი

```ts
export class BcryptProvider extends HashingProvider {
  async hash(value: string): Promise<string> {
    return bcrypt.hash(value, 10);
  }

  async compare(value: string, hashedValue: string): Promise<boolean> {
    return bcrypt.compare(value, hashedValue);
  }
}
```

ეს უკვე აკეთებს რეალურ საქმეს (პაროლის დაშიფვრას და შედარებას).

---

### 🧠 რატომ არ ვიყენებთ პირდაპირ bcrypt-ს?

თუ პირდაპირ ყველგან ჩაწერ:

```ts
bcrypt.hash(password, 10);
```

მერე როცა გადაწყვიტავ bcrypt-ის ნაცვლად სხვა ბიბლიოთეკა გამოიყენო (მაგ. Argon2), მოგიწევს ყველა ფაილის შეცვლა 😩
ეს ცუდია.

მაგრამ თუ შენ იყენებ "გეგმას" (HashingProvider) → შეგიძლია მარტივად შეცვალო ერთი ხაზით და იმუშავებს ისე, თითქოს არაფერი შეცვლილა 🤩

---

### 🪄 როგორ მუშაობს ეს Nest-ში

Nest-ს ვეუბნებით:

```ts
{
  provide: HashingProvider,
  useClass: BcryptProvider,
}
```

ანუ:

> როცა ვინმეს სჭირდება HashingProvider — მიეცი მას BcryptProvider.

ხვალ თუ სხვა გინდა:

```ts
{
  provide: HashingProvider,
  useClass: ArgonProvider,
}
```

და ყველაფერი იმუშავებს იგივე კოდით 💪

### 🧩 რატომ აბსტრაქტული კლასი და არა interface?

- TypeScript-ის **interface** მხოლოდ კოდის დროს არსებობს (შემოწმებისთვის)
- მაგრამ Nest-ს სჭირდება რაღაც **ნამდვილი ობიექტი** runtime-ზე (როცა აპი გაშვებულია).
  Interface იქ აღარ არსებობს ❌
- აბსტრაქტული კლასი კი **არსებობს** და შეიძლება გამოვიყენოთ როგორც token. ✅

## 🧩 1. შენ თუ გააკეთებდი ჩვეულებრივ კლასს

მაგალითად ასე:

```ts
export class HashingProvider {
  async hash(value: string): Promise<string> {
    return bcrypt.hash(value, 10);
  }

  async compare(value: string, hashedValue: string): Promise<boolean> {
    return bcrypt.compare(value, hashedValue);
  }
}
```

და მერე უბრალოდ inject გექნებოდა:

```ts
@Injectable()
export class AuthService {
  constructor(private readonly hashing: HashingProvider) {}
}
```

👉 ეგ იმუშავებს!
შენი კოდი იმუშავებს **სწორად**, bcrypt-ით hash-ებს გააკეთებს, პაროლებს შეადარებს და ა.შ.
ანუ — **ტექნიკურად პრობლემა არ არის.**

---

## ❗ მაგრამ აქ არის პრობლემა დიზაინში, არა კოდში

როცა ასე წერ:

```ts
constructor(private readonly hashing: HashingProvider)
```

შენი სერვისი უკვე **მიბმულია ამ კონკრეტულ კლასზე.**
ანუ AuthService → პირდაპირ ეყრდნობა `HashingProvider` კლასს, სადაც bcrypt წერია.
თუ ერთ დღეს მოინდომებ bcrypt-ის ნაცვლად Argon2 ან სხვა ალგორითმის გამოყენებას,
შენ მოგიწევს შეცვალო `HashingProvider`-ის შიგთავსი, ან საერთოდ სხვა ფაილში გადაკეთება 😩

ეს ნიშნავს — შენს კოდში **მიმაგრებული ხარ კონკრეტულ შესრულებაზე** (tight coupling).

## 🧱 2. აბსტრაქტ კლასის იდეა

აბსტრაქტ კლასი არის როგორც “წესი”, რომელსაც სხვები უნდა დაემორჩილონ.

```ts
export abstract class HashingProvider {
  abstract hash(value: string): Promise<string>;
  abstract compare(value: string, hashedValue: string): Promise<boolean>;
}
```

ეს არ აკეთებს არაფერს — უბრალოდ ამბობს:

> ვინც იქნება ჰეშინგის პროვაიდერი, ვალდებულია ქონდეს `hash` და `compare`.

ახლა შეგიძლია გააკეთო:

- `BcryptProvider` → bcrypt-ზე დაფუძნებული ვერსია
- `Argon2Provider` → argon-ზე დაფუძნებული ვერსია
- `FakeHashingProvider` → ტესტებისთვის (უბრალოდ აბრუნებს იგივე value-ს)

და AuthService ამას ვერ მიხვდება 😎
ის მხოლოდ ხედავს აბსტრაქტულ კლასს და უბრალოდ იყენებს.
